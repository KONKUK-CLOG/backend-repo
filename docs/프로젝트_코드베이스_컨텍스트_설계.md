# 프로젝트 코드베이스 컨텍스트 설계

이 문서는 유저·프로젝트·채팅 세션 계층과 Lambda 페이로드 확장, 세션 rotation 정책을 정리한 설계 기록이다.

## 배경

- 기존 `ChatSession`은 `userId`만으로 “어떤 코드베이스 맥락인지”를 구분할 수 없었다.
- 동일 프로젝트 내 여러 채팅 세션이 같은 파일 스냅샷을 공유해야 LLM·블로그 생성의 일관성을 유지할 수 있다.

## 데이터 구조 (MongoDB + 연관)

```text
User (MySQL)
  └── Project (Mongo: projects)
        ├── ProjectFile (Mongo: project_files) — filePath, language, content
        └── ChatSession (Mongo: chat_sessions) — userId, projectId(optional), …
              └── ChatMessage (Mongo: chat_messages) — sessionId, role, content, codeSnippets, …
```

- **프로젝트 미연동(레거시)**: `ChatSession.projectId == null`. 활성 세션은 `(userId, projectId=null)` 기준으로 조회한다.
- **토큰 예산**: 프로젝트 파일 전체를 매 요청마다 실으므로, 파일 개수·크기 상한은 `application.properties`로 제한한다.

## 세션 rotation (토큰 한도 초과 시)

1. Lambda `SUMMARIZE` 호출로 대화 요약을 받는다.
2. 요약문을 `systemMessage`로 가진 **새 ACTIVE 세션**을 저장한다 (`projectId` 유지).
3. 기존 세션의 메시지 전체 삭제 후 기존 세션 문서를 삭제한다.
4. 이어서 같은 요청에서 Lambda `GENERATE`를 호출한다.

한도 미만이면 `GENERATE`만 1회 호출한다.

설정 키:

- `app.chat.max-context-tokens` (기본 100000, 환경변수 `APP_CHAT_MAX_CONTEXT_TOKENS`)

## Lambda 입력 계약 (`LambdaPayload`)

`GENERATE` 시 선택 필드:

```json
{
  "action": "GENERATE",
  "userId": 1,
  "chatHistory": [],
  "codeSnippets": [],
  "prompt": "...",
  "projectFiles": [
    { "filePath": "src/main.py", "language": "python", "content": "..." }
  ]
}
```

- 세션에 `projectId`가 있으면 백엔드가 `project_files`에서 읽어 `projectFiles`에 채운다.
- Lambda 구현체는 이 필드를 무시해도 되지만, 코드베이스 맥락을 쓰려면 파싱·프롬프트 주입이 필요하다.

## REST API (`/api/projects`)

| Method | Path | 설명 |
|--------|------|------|
| POST | `/api/projects` | 프로젝트 생성 |
| GET | `/api/projects` | 내 프로젝트 목록 |
| DELETE | `/api/projects/{projectId}` | 프로젝트 삭제 (파일·해당 프로젝트 채팅 세션·메시지 정리) |
| POST | `/api/projects/{projectId}/files` | 파일 추가 |
| GET | `/api/projects/{projectId}/files` | 파일 목록 |
| PUT | `/api/projects/{projectId}/files/{fileId}` | 파일 수정 |
| DELETE | `/api/projects/{projectId}/files/{fileId}` | 파일 삭제 |

채팅 API는 `ChatSendRequest.projectId`(및 선택적 `chatSessionId`), `GET /api/chat/history?projectId=` 로 프로젝트 스코프를 맞춘다.

## 파일 업로드 한도

- `app.project.max-files-per-project` (`APP_PROJECT_MAX_FILES`, 기본 20)
- `app.project.max-file-bytes` (`APP_PROJECT_MAX_FILE_BYTES`, 기본 51200)

## 보안

- `/api/projects/**`는 JWT 인증이 필요하다 (`SecurityConfig`의 기본 규칙에 포함).

## 관련 코드 위치

- 채팅·페이로드: `konkuk.clog.domain.chat.service.ChatService`, `konkuk.clog.domain.llm.dto.LambdaPayload`, `ProjectFileContext`
